# 智能体系统 v2 — 整体设计文档

## 一、核心原则

| # | 原则 | 说明 |
|---|------|------|
| 1 | 模块化+单文件 | 架构可拆可合，短场景单文件运行，长场景模块拆分 |
| 2 | 全JSON配置驱动 | 用户只写配置，不写代码、不改代码 |
| 3 | 低耦合大框架 | 各模块独立运行，互不依赖，互不通信 |
| 4 | 可扩展可插拔 | 接口即插即用，热插拔新增功能 |
| 5 | 自定义LLM规则 | 与大模型交互的prompt、格式、规则全部用户自定义 |
| 6 | 全透明 | LLM原始输入输出完全可见，不做封装、过滤、隐藏 |
| 7 | 前端化易安装 | 简单Web界面，一键启动，全球平台可分发 |
| 8 | 零依赖复制即用 | 无外部pip包依赖，全本地路径，复制文件夹即可运行 |
| 9 | 全中文命名 | 函数、变量、文件夹、界面全中文，方便中国人开发 |
| 10 | 全局/局部双层 | 类似Unity：全局命令+全局事件中心，局部文件挂载模块，局部事件中心可引入（后期） |
| 11 | 运行时高度实时 | 界面与大模型联动，大模型可看人在干什么，后期可实时操作电脑 |
| 12 | 双引擎自我进化 | 主引擎稳定发布+工作引擎大模型开发，人决定何时合并 |
| 13 | 进化安全红线 | 主引擎不可自改，权限不可自改，检测不过不合并，失败自动暂停 |
| 14 | 文件安全铁律 | 操作任何新文件夹必须问用户，无权限绝对不动，3次放行可升永久，权限存JSON随时改 |
| 15 | 大模型全权决策 | 系统不自己设计思路，所有问题丢给大模型，由大模型负责优化、解答、决策 |
| 16 | 数字资产原则 | 一切代码和数据都是数字资产，尊重每一行代码，不允许冗余，为未来复用打基础 |
| 17 | 日志+Git铁律 | 每次开发写好日志，每次提交Git含日期时间+标题，完整可追溯 |
| 18 | 隐私分区 | 隐私区域与公开区域严格区分，隐私不进发布包，一键打包只打公开区 |
| 19 | 相对路径优先 | 所有路径使用相对路径，禁止硬编码绝对路径，复制到任何位置都能跑 |

## 四、内核设计（极小，仅4个职责）

### 4.1 启动器
- 读取 `系统配置.json`，确定运行模式（模块化/单文件）
- 加载配置，启动Web服务
- **不做任何业务逻辑**

### 4.2 配置加载器
- 纯JSON读取，零解析框架
- 支持配置热重载（监听文件变化）
- 所有行为由配置决定，代码无硬编码

### 4.3 模块调度器
- 扫描 `模块/` 目录，读取每个模块的 `模块声明.json`
- 按配置启用/禁用模块
- 模块间**不通信、不依赖**，调度器只负责加载和卸载
- 每个模块是一个独立Python文件，通过统一接口注册

### 4.4 模型直连器
- **纯HTTP请求**，使用Python内置 `urllib` 或 `http.client`
- 不依赖openai、anthropic等SDK
- 用户在 `模型规则.json` 中自定义：
  - API地址（支持任意兼容端点）
  - 请求头格式
  - 请求体格式
  - 响应解析规则（JSONPath表达式）
  - System Prompt模板
  - 对话格式规则
- **全透明**：每次LLM调用，原始请求和响应写入日志文件，Web界面可实时查看

### 4.5 文件管理器
- 负责所有文件的读写操作，**智能体不可直接访问文件系统**
- 每次读写必须经过权限校验（读取 `文件权限.json`）
- 支持操作：读文件、写文件、列目录、创建目录、删除、移动/重命名
- 每次操作记录审计日志（谁、何时、做了什么）
- Web界面的编辑器通过文件管理器读写文件，用户可实时查看变更

### 4.6 全局命令中心
- 类似Unity的全局静态入口，任何模块/插件可直接调用
- 提供系统级指令：`退出` `重启` `重载配置` `开关模块` `发送全局事件`
- 命令通过 `全局命令.json` 注册，用户可自定义新命令

```python
# 全局命令调用方式（约定）
全局命令.执行("重载配置")
全局命令.执行("开关模块", {"模块名": "对话", "状态": "禁用"})
```

### 4.7 全局事件中心
- 发布/订阅模式，模块可监听全局事件，但**不产生模块间依赖**
- 内置事件：`系统启动` `系统关闭` `模块加载` `模块卸载` `配置变更` `文件变更`
- 用户可在 `事件规则.json` 中自定义事件和触发条件

```python
# 全局事件使用方式（约定）
全局事件.订阅("配置变更", 我的回调函数)
全局事件.发布("文件变更", {"路径": "xxx", "操作": "写入"})
```

### 4.8 运行时实时总线（后期扩展）
- 连接界面与大模型的实时通道
- 用户在界面的操作（点击、输入、滚动）可实时推送至大模型
- 大模型可向界面推送指令（高亮、提示、自动填入）
- 预留接口，当前仅定义协议，不实现

```json
{
  "实时事件": [
    {"来源": "界面", "类型": "用户输入", "数据": "..."},
    {"来源": "界面", "类型": "用户点击", "数据": "..."},
    {"来源": "模型", "类型": "高亮提示", "数据": "..."},
    {"来源": "模型", "类型": "自动填入", "数据": "..."}
  ]
}
```

## 六、模块设计（各干各事）

### 6.1 模块统一接口
每个模块实现以下最小接口（约定，非继承）：

```python
def 初始化(配置): ...        # 初始化，接收模块配置
def 运行(输入数据): ...     # 执行，返回结果
def 停止(): ...              # 停止/清理
```

### 6.2 局部文件挂载（类似Unity组件挂载）
- 每个模块可挂载专属文件/目录，这些文件**只属于该模块**
- 挂载在 `模块声明.json` 中声明，模块销毁时可选择是否清理挂载文件
- 其他模块**不可访问**其他模块的挂载文件（隔离）

```json
{
  "名称": "对话",
  "挂载文件": [
    {"路径": "./模块/对话/对话历史.json", "权限": "读写", "清理时删除": false},
    {"路径": "./模块/对话/临时缓存/", "权限": "读写", "清理时删除": true}
  ]
}
```

### 6.3 局部事件中心（后期引入）
- 每个模块内部可有自己的局部事件中心
- 局部事件只在模块内部传播，不污染全局
- 模块可选择将局部事件**冒泡**到全局事件中心

```python
# 局部事件（后期）
局部事件 = 创建局部事件中心()

局部事件.订阅("消息到达", 处理消息)
局部事件.发布("消息到达", {"内容": "你好"})

# 冒泡到全局（可选）
局部事件.冒泡("消息到达")  # 此事件同时发送到全局事件中心
```

### 6.4 模块间通信
**不通信。** 模块之间零联系。唯一的中转方式：
- **全局事件中心**：A模块发布事件，B模块订阅（松耦合，不直接调用）
- **JSON文件中转**：A模块写JSON，B模块读JSON
- **不推荐直接调用**，违反低耦合原则

## 七、记忆系统（核心设计）

> **一段对话对应一件事，不忘记，摘要重点，清理冗余，让大模型知道你是怎么想的。**
> **全部使用JSON存储，零数据库依赖，复制即用。**

### 7.1 设计理念

传统对话记忆是一条条消息平铺，越聊越长，大模型丢失上下文。
本系统采用**事件制记忆**：每次对话归类为一个"事件"，事件有完整生命周期，
不会淹没在消息流中，大模型可以精确回忆"当时那件事"。

```
传统记忆:  [消息1][消息2][消息3][消息4][消息5]...  ← 越长越模糊
本系统:    [事件A: 完整经过] [事件B: 完整经过] [事件C: ...]  ← 一件事一个盒子
                ↓摘要              ↓摘要
           [摘要A]            [摘要B]              ← 需要时只看摘要
```

### 7.2 记忆三层结构

```
┌─────────────────────────────────────────────┐
│           第一层：用户画像（长期）            │
│  用户思维模式 / 偏好 / 习惯 / 性格特征        │
│  大模型通过此层理解"用户是怎么想的"           │
├─────────────────────────────────────────────┤
│           第二层：事件记忆（中期）            │
│  每段对话=一个事件，完整保留始末              │
│  不删除原始记录，只压缩冗余                   │
├─────────────────────────────────────────────┤
│           第三层：当前上下文（短期）           │
│  当前正在聊的事件，原始对话全文               │
│  聊完归档到第二层                             │
└─────────────────────────────────────────────┘
```

### 7.3 全JSON记忆存储结构

**所有记忆存储为JSON文件，无数据库、无二进制、无特殊格式。**
用户可直接用文本编辑器打开查看和修改，完全透明。

```
模块/记忆/
├── 记忆库.json              ← 全部记忆集中于此，一个JSON管所有
├── 用户画像.json            ← 用户思维模式画像
└── 摘要索引.json            ← 快查索引（事件编号→摘要一句话）
```

### 7.7 记忆注入策略（长上下文适应）

大模型每次请求时，记忆按优先级注入系统提示词：

```
┌──────────────────────────────────────┐
│           注入优先级                  │
│                                      │
│  1. 用户画像         ← 必注入，量小   │
│     (思维模式+偏好)     ~200 token   │
│                                      │
│  2. 相关事件摘要      ← 按标签匹配   │
│     (当前话题相关的)    ~500 token   │
│                                      │
│  3. 当前事件原始对话  ← 正在聊的事件  │
│     (完整上下文)        ~1000 token  │
│                                      │
│  4. 近期事件摘要      ← 最近N件事    │
│     (补充背景)          ~300 token   │
│                                      │
│  总计约 2000 token，适配长上下文      │
│  超出时按优先级从4→1截断             │
└──────────────────────────────────────┘
```

注入模板（用户可在 `模型规则.json` 中自定义）：

```json
{
  "记忆注入模板": "【用户画像】\n${用户画像}\n\n【相关记忆】\n${相关事件摘要}\n\n【当前对话】\n${当前事件原始}\n\n【近期摘要】\n${近期事件摘要}"
}
```

### 7.8 摘要生成流程

```
事件对话进行中
    │
    ├── 对话超过20轮 → 触发中间摘要（压缩当前上下文窗口）
    │
    └── 用户结束话题 / 切换话题 → 触发归档摘要
            │
            ▼
    大模型接收: 原始对话全文 + 用户画像
    大模型输出: 摘要JSON（核心内容+关键决策+用户意图+待办）
            │
            ▼
    写入 记忆库.json 对应事件的"摘要"字段
    更新 用户画像.json（如有新偏好发现）
    更新 摘要索引.json
```

### 7.9 冗余清理策略

```
定期扫描（配置触发，默认每天）
    │
    ├── 扫描 记忆库.json 中每个事件的"原始对话"
    │       │
    │       ├── 标记1: 重复信息（用户反复确认同一内容）→ 打冗余标记
    │       ├── 标记2: 闲聊寒暄（与事件主题无关的对话）→ 打冗余标记
    │       └── 标记3: 错误尝试（已被纠正的错误路径）→ 打冗余标记
    │
    ├── 清理规则
    │       ├── 原始对话永不物理删除（只打标记）
    │       ├── 注入上下文时跳过冗余标记的消息
    │       ├── 摘要已覆盖的内容，原始对话可选压缩
    │       └── 超过阈值天数且无关联的事件，摘要保留，原始对话归档压缩
    │
    └── 用户可在界面手动确认清理结果
```

### 7.10 JSON记忆分片策略

当 `记忆库.json` 过大时（超过配置阈值），自动分片：

```json

{
  "版本": "1.0",
  "当前事件": "事件_003",
  "分片列表": [
    {"文件": "记忆库_001.json", "事件范围": ["事件_001", "事件_050"], "大小KB": 85},
    {"文件": "记忆库_002.json", "事件范围": ["事件_051", "事件_100"], "大小KB": 72}
  ],
  "全局设置": {
    "分片大小阈值KB": 100,
    "每片最大事件数": 50
  }
}
```

分片规则：
- 默认每个分片不超过100KB或50个事件
- 当前进行中的事件始终在主文件中
- 分片文件按需加载，不全部读入内存
- 合并时只需将分片JSON合并回 `事件列表` 即可

- 用户在Web界面浏览本地目录树
- 一键授权/撤销智能体对目录的访问权限
- 实时查看智能体读写文件的审计记录
- **任何未授权文件夹操作必须询问用户**
- **绝对不能动无权限的文件**

### 12.2 目录树浏览
- 左侧侧边栏显示目录树（懒加载，点击展开）
- 文件夹图标📁/文件图标📄，按类型区分颜色
- 权限状态实时显示：✅永久授权 / 🔶仅一次 / ❌禁止 / ⚪未授权
- 右键菜单：打开编辑器、授权访问、撤销权限、查看属性

### 12.3 权限管理
```
┌──────────────────────────────────────────────────────┐
│  文件管理                                             │
├──────────┬───────────────────────────────────────────┤
│ 📁 D:/   │  📁 我的项目/                              │
│ ├📁 用户  │  ├── 📁 数据/    ✅ 永久授权(3次后自动升级) │
│ ├📁 程序  │  ├── 📁 配置/    🔶 仅一次授权             │
│ └📁 我的  │  ├── 📁 日志/    ❌ 禁止                   │
│   项目/   │  ├── 📁 临时/    ⚪ 未授权(0次询问)        │
│           │  └── 📄 说明.md   ✅ 永久可读写              │
├──────────┴───────────────────────────────────────────┤
│  [授权选中目录]  [撤销权限]  [改为永久]  [刷新]         │
│                                                      │
│  ⚠ 询问弹窗（智能体请求访问未授权路径时）：             │
│  ┌────────────────────────────────────────────┐      │
│  │  智能体请求访问: D:/我的项目/临时/           │      │
│  │  操作: 读取                                 │      │
│  │  已询问此路径: 1次 (再2次自动永久放行)       │      │
│  │                                            │      │
│  │  [✅ 允许一次]  [🔒 永久允许]  [❌ 拒绝]     │      │
│  └────────────────────────────────────────────┘      │
│                                                      │
│  操作记录:                                            │
│  00:15 智能体读取 D:/我的项目/数据/a.json  ✅          │
│  00:14 智能体写入 D:/我的项目/配置/b.json  ✅          │
│  00:13 智能体读取 D:/系统目录/x.json       ❌ 拒绝     │
│  00:12 用户授权 D:/我的项目/数据/ 永久权限             │
└──────────────────────────────────────────────────────┘
```

### 12.4 权限校验流程（核心安全机制）

```
智能体请求文件操作
    │
    ▼
文件管理器拦截
    │
    ├── 检查1: 路径是否在"禁止关键词路径"中？
    │       └── 是 → 直接拒绝，不询问，记录拒绝日志
    │
    ├── 检查2: 路径是否在"授权目录"中？
    │       ├── 是，且"授权类型"="永久" → 执行 + 记录审计日志
    │       ├── 是，且"授权类型"="一次" → 执行本次 + 记录 + 自动移除本次授权
    │       └── 是，且"授权类型"="禁止" → 直接拒绝，不询问
    │
    ├── 检查3: 路径未在"授权目录"中（新路径）
    │       │
    │       ▼
    │   弹窗询问用户（三种选择）
    │       ├── [允许一次] → 本次执行 + 记录到询问记录.json + 询问次数+1
    │       │                    └── 询问次数≥3 → 自动升级为永久授权 + 通知用户
    │       ├── [永久允许] → 执行 + 写入文件权限.json为永久授权
    │       └── [拒绝]   → 不执行 + 记录拒绝日志 + 询问次数不变
    │
    └── 绝对原则: 无权限 = 绝对不动，没有任何绕过方式
```

### 12.5 三级授权类型

| 类型 | 说明 | 生命周期 |
|------|------|----------|
| **一次** | 仅本次操作有效，操作完即失效 | 单次操作 |
| **永久** | 持续有效，直到用户主动撤销 | 永久 |
| **禁止** | 绝对不允许，连询问都不弹 | 永久（直到用户改） |

### 12.6 询问3次自动升级规则

```json
{
  "规则": {
    "同一路径询问达到3次": "自动升级为永久授权",
    "升级前通知用户": true,               // 升级时弹窗告知用户
    "升级后用户可改回": true,              // 用户可随时撤销或降级
    "询问计数跨重启保留": true,            // 询问记录持久化到JSON
    "用户主动撤销后重新计数": true          // 撤销后计数归零，重新累积
  }
}
```

升级流程：
```
第1次询问 → 用户选"允许一次" → 询问记录+1 (1/3)
第2次询问 → 用户选"允许一次" → 询问记录+1 (2/3)
第3次询问 → 用户选"允许一次" → 询问记录+1 (3/3)
    │
    ▼
自动升级为永久授权
    │
    ▼
通知用户: "D:/我的项目/临时/ 已询问3次，自动升级为永久授权，可随时撤销"
    │
    ▼
写入 文件权限.json (授权类型改为"永久")
清零 询问记录.json 中此路径的计数
```

### 12.7 用户随时改动权限

用户可随时在界面执行以下操作：

| 操作 | 说明 |
|------|------|
| 永久→一次 | 降级为仅一次授权 |
| 永久→禁止 | 撤销权限，禁止访问 |
| 一次→永久 | 手动升级为永久 |
| 一次→禁止 | 拒绝后续访问 |
| 禁止→永久 | 解除禁止，授权永久 |
| 禁止→一次 | 解除禁止，仅允许一次 |
| 删除路径 | 从授权目录中移除（变回未授权状态） |

所有权限变更立即写入 `文件权限.json`，实时生效。

### 12.8 审计日志
所有文件操作记录到 `日志/文件操作.json`：
```json
{
  "时间": "2026-06-17T00:15:00",
  "操作者": "智能体",
  "操作": "写入",
  "路径": "D:/我的项目/数据/a.json",
  "结果": "成功",
  "变更摘要": "修改了3行，新增了1行"
}
```

## 十六、双引擎架构（主引擎+工作引擎）

### 16.1 核心理念

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│   公共区 🔵（主引擎）            隐私区 🔴（工作引擎）       │
│   ┌───────────────┐            ┌───────────────┐           │
│   │  稳定可用      │  ←复制←    │  大模型开发    │           │
│   │  用户日常使用  │  人决定    │  自我进化实验  │           │
│   │  只接受合并    │            │  随便改随便试  │           │
│   │  可打包发布    │            │  绝不进发布包  │           │
│   └───────────────┘            └───────────────┘           │
│         ↑                              ↑                    │
│     人决定何时复制                 主引擎有权管理             │
│                                                             │
│   🔵 公共区 = 引擎+界面+公共配置（可开源/可分发）            │
│   🔴 隐私区 = 密钥+记忆+数据+日志+工作引擎（绝不外泄）      │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

- **主引擎**：用户使用的稳定版本，不接受直接修改，只接受从工作引擎复制过来的验证通过的代码
- **工作引擎**：大模型负责开发的版本，可以随意修改、实验、迭代
- **复制权在人**：只有人决定什么时候把工作引擎的改动复制到主引擎
- **主引擎管工作引擎**：主引擎有权查看、管理、重置工作引擎的任何内容

- 主引擎代码**大模型不可直接修改**，只能通过合并流程从工作引擎复制

- 连续失败超过阈值自动暂停，需人工恢复
- 每次合并前自动备份主引擎，可一键回滚

## 十七、大模型全权决策（系统不做大脑）

### 17.1 核心原则

> **系统是手脚，大模型是大脑。系统不自己想，一切丢给大模型。**

传统智能体框架：系统内建逻辑流程、条件判断、路由规则 → 大模型只是执行器
本系统：系统只负责调度和搬运 → 大模型负责一切思考、决策、优化

```
传统框架:  用户→系统逻辑判断→分支→调用大模型→系统再判断→...
本系统:    用户→系统原样搬运→大模型思考决策→系统原样执行→大模型再思考→...
```

### 17.2 系统 vs 大模型的职责边界

| 职责 | 系统（内核/模块） | 大模型 |
|------|-------------------|--------|
| 对话意图理解 | ❌ 只搬运消息 | ✅ 理解、分析、回复 |
| 摘要生成策略 | ❌ 只搬运对话给模型 | ✅ 决定摘要什么、怎么摘 |
| 冗余判断 | ❌ 只搬运对话给模型 | ✅ 判断哪些冗余、打标记 |
| 用户画像更新 | ❌ 只搬运历史给模型 | ✅ 分析用户特征、提炼偏好 |
| 模块调度 | ✅ 按配置加载/卸载 | ✅ 决定何时启用/禁用模块 |
| 文件权限 | ✅ 硬性拦截+询问 | ❌ 无权绕过权限 |
| 代码自我优化 | ❌ 不自己改代码 | ✅ 在工作引擎中优化 |
| 异常处理思路 | ❌ 不自己决定怎么办 | ✅ 把错误丢给模型，让它决定 |
| 事件关联分析 | ❌ 不自己判断关联 | ✅ 分析事件标签、建立关联 |
| 任务分解 | ❌ 不自己拆任务 | ✅ 大模型拆解、编排、执行 |

### 17.3 全权决策工作流

```
任何模块遇到需要"思考"的场景
    │
    ▼
不自己设计逻辑，而是打包问题上下文
    │
    ▼
通过模型直连器 发送给大模型
    │
    ├── 问题类型: 摘要 / 判断 / 优化 / 决策 / 分析 / 修复
    ├── 附带上下文: 相关记忆 + 当前状态 + 用户画像
    └── 附带约束: 文件权限规则 + 禁止操作 + 安全红线
    │
    ▼
大模型返回决策结果（JSON格式）
    │
    ▼
系统原样执行，不二次判断
    │
    ▼
执行结果反馈给大模型，大模型决定是否继续
```

### 17.5 系统硬性规则（不可丢给大模型的）

以下规则**硬编码在系统中**，大模型无权决定：

| 硬性规则 | 原因 |
|----------|------|
| 文件权限拦截 | 安全红线，不可由模型绕过 |
| 禁止路径检查 | 系统目录绝对不可碰 |
| 合并需人工确认 | 主引擎不可自动合并 |
| 自我进化禁止范围 | 核心文件不可自改 |
| 权限配置不可模型自改 | 防提权 |
| 连续失败自动暂停 | 防止失控循环 |

### 17.6 设计哲学

> **系统笨一点没关系，大模型聪明就行。**
>
> 系统不自己思考，就不会想错。
> 系统只做搬运工和执行者，大模型做大脑。
> 遇到问题不自己设计思路，打包丢给大模型。
> 大模型的决策全透明，用户随时查看和干预。

## 十八、数字资产管理

### 18.1 核心原则

> **一切都是数字资产。每一行代码、每一个配置、每一段记忆，都是可复用的资产。**
> **尊重每一行代码，不允许冗余，为未来利用做好基础。**

### 18.2 资产分类

| 资产类型 | 存储位置 | 说明 |
|----------|----------|------|
| 代码资产 | 内核/、模块/、插件/ | 每个函数都有存在价值，无冗余 |
| 配置资产 | 配置/ | 全JSON，人类可直接阅读修改 |
| 记忆资产 | 模块/记忆/ | 对话即资产，摘要即索引，永不丢失 |
| 日志资产 | 日志/ | 每次操作可追溯，每次决策有据可查 |
| 开发日志 | 开发日志.md | 每次开发记录：做了什么、删了什么、为什么 |
| Git历史 | .git/ | 完整版本时间线，随时回溯任意版本 |

### 18.3 反冗余规则

```
编写代码前：
    ├── 这行代码是否已有同等功能存在？ → 有 → 不写，复用
    ├── 这个模块是否和已有模块职责重叠？ → 重叠 → 合并，不新建
    ├── 这个配置是否和已有配置重复？ → 重复 → 引用，不复写
    └── 这个文件是否可以不存在？ → 可以 → 不创建

编写代码后：
    ├── 有无未使用的函数？ → 有 → 删除
    ├── 有无重复的逻辑？ → 有 → 提取公共函数
    ├── 有无硬编码的值？ → 有 → 移到配置JSON
    └── 有无可删除的注释？ → 有 → 删（代码即文档）
```

### 18.4 资产复用机制

```json
{
  "资产注册表": [
    {"类型": "函数", "名称": "读取JSON", "位置": "内核/配置加载器.py:15", "被引用": 12},
    {"类型": "函数", "名称": "权限校验", "位置": "内核/文件管理器.py:30", "被引用": 8},
    {"类型": "配置", "名称": "模型规则", "位置": "配置/模型规则.json", "被引用": 3},
    {"类型": "模块", "名称": "记忆", "位置": "模块/记忆/", "被引用": 2}
  ]
}
```

- 新开发前先查资产注册表，有则复用
- 新资产写入注册表，标记位置和引用关系
- 删除资产前检查引用，有引用则禁止删除

## 十九、开发日志与Git管理

### 19.1 开发日志（开发日志.md）

每次开发**必须写日志**，记录精确时间、做了什么、删了什么。

```markdown
# 开发日志

## 2026-06-17 14:30
- [新增] 内核/文件管理器.py — 权限校验+审计+3次放行逻辑
- [新增] 配置/询问记录.json — 路径询问计数持久化
- [修改] 配置/文件权限.json — 新增授权类型字段（一次/永久/禁止）
- [删除] 内核/旧工具.py — 功能已合并到文件管理器，不再需要
- [修复] 模块/记忆/主程序.py — 修复摘要生成时索引越界

## 2026-06-17 10:00
- [新增] 内核/启动器.py — 初始版本
- [新增] 内核/配置加载器.py — 初始版本
- [新增] 内核/模型直连器.py — 初始版本
- [新增] 配置/系统配置.json — 初始配置
```

日志格式规则：
- 每条记录必须包含**精确到分钟的时间**
- 每条记录标注类型：`[新增]` `[修改]` `[删除]` `[修复]`
- 删除的东西必须记录**为什么删**
- 不允许空日志提交

### 19.2 Git提交铁律

每次Git提交**必须包含日期时间和标题**：

```
格式: [类型] YYYY-MM-DD HH:MM 标题内容

示例:
[新增] 2026-06-17 14:30 文件管理器权限校验+3次放行
[修复] 2026-06-17 10:00 记忆摘要索引越界
[优化] 2026-06-17 09:30 冗余清理策略调整
[合并] 2026-06-17 16:00 工作引擎→主引擎 v1.0.1
[重构] 2026-06-17 08:00 配置加载器拆分为独立模块
```

类型列表：

| 类型 | 说明 |
|------|------|
| 新增 | 新功能、新文件、新模块 |
| 修复 | 修bug |
| 优化 | 性能/逻辑优化，功能不变 |
| 合并 | 工作引擎合并到主引擎 |
| 重构 | 结构调整，功能不变 |
| 配置 | 仅修改JSON配置 |
| 删除 | 删除文件/功能 |

### 19.3 提交流程

```
开发完成
    │
    ├── 1. 检查冗余 → 有冗余则先清理
    ├── 2. 写开发日志 → 记录本次所有变更
    ├── 3. git add . → 全部暂存（不漏文件）
    ├── 4. git commit → "[类型] YYYY-MM-DD HH:MM 标题"
    └── 5. 确认提交成功 → git status
```

### 19.4 Git提交规范（写入系统配置）

```json
{
  "Git规范": {
    "提交格式": "[类型] YYYY-MM-DD HH:MM 标题",
    "类型列表": ["新增", "修复", "优化", "合并", "重构", "配置", "删除"],
    "提交前必须写日志": true,
    "提交前检查冗余": true,
    "全量提交": true,
    "禁止推送远程": "未经用户确认"
  }
}
```

### 19.5 版本追溯

```
任何一行代码的来历：
    → Git blame 看谁写的、什么时候写的
    → Git log 看那次提交的完整变更
    → 开发日志 看当时为什么这么改
    → 记忆库 看当时的需求上下文
```

所有数字资产，从代码到配置到记忆到日志到Git，形成完整追溯链。

## 二十、隐私分区与一键发布

### 20.1 核心理念

> **公共区是脸面，隐私区是里子。发布只发脸面，里子绝不外泄。**

```
智能体_v2/
  │
  ├── 【公共区】/ 🔵  →  可以开源 / 可以发GitHub / 可以打包给任何人
  │                    →  不含任何个人数据、密钥、隐私
  │
  └── 【隐私区】/ 🔴  →  绝对不外传 / 不进发布包 / 不进Git公开仓库
                       →  股票数据、个人记忆、API Key、工作日志...
```

### 20.2 分区对照表

| 内容 | 公共区 🔵 | 隐私区 🔴 | 说明 |
|------|----------|----------|------|
| 内核代码 | ✅ | - | 引擎是公共的 |
| 模块代码 | ✅ | - | 功能模块是公共的 |
| 界面代码 | ✅ | - | 前端是公共的 |
| 公共配置 | ✅ | - | 系统配置、模块配置、事件规则 |
| 工作引擎 | - | ✅ | 开发版引擎，含实验代码 |
| API Key | - | ✅ | 绝不进公共区 |
| 对话记忆 | - | ✅ | 含个人对话内容 |
| 用户画像 | - | ✅ | 个人思维模式，绝对隐私 |
| 运行日志 | - | ✅ | 含LLM原始输入输出 |
| 询问记录 | - | ✅ | 文件权限询问历史 |
| 个人数据 | - | ✅ | 股票数据、项目文档等 |
| 插件代码 | ✅ | - | 公共插件进公共区 |
| 私有插件 | - | ✅ | 个人专用插件进隐私区 |

### 20.3 隐私区目录详解

```
【隐私区】/ 🔴
│
├── 【我的数据】/                    ← 个人工作数据
│   ├── 股票数据/                   # 量化交易、行情数据
│   ├── 项目文档/                   # 私人项目资料
│   ├── 财务记录/                   # 个人财务
│   └── (用户自定义)
│
├── 【我的记忆】/                    ← 全JSON个人记忆
│   ├── 记忆库.json                 # 含隐私对话内容
│   ├── 记忆库_001.json
│   ├── 用户画像.json               # 个人思维模式
│   └── 摘要索引.json
│
├── 【我的配置】/                    ← 含敏感信息
│   ├── 密钥.json                   # API Key / Token / 密码
│   ├── 询问记录.json               # 权限询问历史
│   └── 私有配置.json               # 个人专属配置覆盖
│
├── 【我的日志】/                    ← 运行日志
│   └── (按日期自动生成，含LLM原始对话)
│
└── 【我的工作引擎】/                ← 开发版引擎
    └── (同公共区结构)
```

### 20.4 密钥.json（绝不进公共区）

```json
{
  "说明": "所有敏感密钥存于此文件，绝不进入公共区",
  "密钥列表": {
    "LLM_API_KEY": "sk-xxxxxxxxxxxxx",
    "LLM_MODEL": "gpt-4",
    "自定义端点": "https://my-proxy.example.com"
  },
  "读取规则": {
    "仅限模型直连器读取": true,
    "界面显示时掩码": true,
    "日志中掩码": true
  }
}
```

### 20.5 一键发布打包

**发布打包.py** 只打包公共区，隐私区完全排除：

```
发布打包流程:
    │
    ├── 1. 读取打包配置
    ├── 2. 扫描 【公共区】/ 全部文件
    ├── 3. 排除 【隐私区】/ 全部文件
    ├── 4. 检查公共区是否引用了隐私区文件 → 有则警告
    ├── 5. 生成发布目录结构
    ├── 6. 复制文件 + 生成zip
    └── 7. 输出: 智能体_v2_发布_v1.0.0.zip
```

打包配置：
```json
{
  "打包配置": {
    "包含目录": ["公共区/"],
    "排除目录": ["隐私区/"],
    "排除文件": [".env", "*.key", "*密钥*"],
    "检查隐私泄露": true,
    "输出格式": "zip",
    "输出命名": "智能体_v2_发布_v${版本号}.zip"
  }
}
```

### 20.6 隐私泄露检查

打包前自动扫描公共区文件，检测是否意外包含隐私内容：

| 检查项 | 规则 |
|--------|------|
| API Key | 检测 `sk-`、`key=`、`token=` 等模式 |
| 绝对路径 | 检测 `D:/`、`C:/Users/` 等个人路径 |
| 邮箱手机 | 检测邮箱格式、手机号格式 |
| 隐私关键词 | 用户自定义关键词列表 |
| 引用隐私区 | 检测代码中是否引用隐私区路径 |

发现任何泄露 → **阻止打包** + 报告具体位置 + 必须修复后重新打包

### 20.7 Git分区策略

```json
{
  "Git策略": {
    "公共区": {
      "可推送远程": true,
      "可公开分享": true,
      ".gitignore": ["隐私区/"]
    },
    "隐私区": {
      "可推送远程": false,
      "仅本地Git": true,
      "独立仓库": true
    }
  }
}
```

- 公共区推送到公开仓库（GitHub等）
- 隐私区用独立本地Git管理，**绝不推送远程**
- `.gitignore` 中排除整个 `隐私区/` 目录

### 20.8 界面中的分区标识

```
┌──────────────────────────────────────────────┐
│  智能体 v2                                    │
├──────────────────────────────────────────────┤
│  🔵 公共区  |  🔴 隐私区  |  ⚙ 引擎管理       │

│  🔴 隐私区 > 我的数据 > 股票数据/            │
```

- 公共区页面顶部显示蓝色标识
- 文件管理器中隐私文件用红色图标标识
- 一键发布按钮只在公共区页面出现

## 二十一、设计哲学

> **够用就行，越小越好。**
> 
> 不追求功能强大，追求结构清晰、配置驱动、透明可控。
> 用户通过JSON定义一切行为，代码只是引擎，配置才是灵魂。
> 一切都是数字资产，尊重每一行代码，不冗余、可追溯、可复用。
> 公共区是脸面可发布，隐私区是里子绝不外泄。
