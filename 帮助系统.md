## 零、30秒快速理解

这是一个**全JSON配置驱动的智能体框架**：
- 系统是手脚（调度、搬运、执行）
- 大模型是大脑（思考、决策、优化）
- 用户是主人（授权、审批、合并）
- 一切可配置、可追溯、可发布、可复用

文件只有两种区域：🔵公共区（可发布）和🔴隐私区（绝不外泄）

### 开源信息
- **GitHub仓库**：https://github.com/zf3dzf3dzf3d-ctrl/zf3d_agent
- **Gitee镜像**：https://gitee.com/zf3d/zf3d_agent
- **下载方式**：`git clone https://gitee.com/zf3d/zf3d_agent.git`（国内推荐Gitee）
- **Linux安装**：clone后运行 `bash install.sh`，或解压发布包后运行 `bash install.sh`
- **网页端口**：8765（http://localhost:8765）
- **当前版本**：v2.3.0
- **许可协议**：MIT License

### 隐私铁律
- `.gitignore` 已排除所有隐私数据（密钥/对话记录/浏览器会话/记忆库/日志/数据库）
- `开发日志.md` 为私有文件，不推送到GitHub（对外用 `CHANGELOG.md`）
- `发布打包.py` 打包前自动扫描隐私泄露（API Key/邮箱/个人路径）
- git历史已清理所有敏感文件记录

## 一、18条核心原则（不可违反）

| # | 原则 | 说明 |
|---|------|------|
| 1 | 模块化+单文件 | 可拆可合，短场景单文件，长场景模块拆分 |
| 2 | 全JSON配置驱动 | 用户只写配置，不写代码、不改代码 |
| 3 | 低耦合大框架 | 各模块独立运行，互不依赖，互不通信 |
| 4 | 可扩展可插拔 | 接口即插即用，热插拔新增功能 |
| 5 | 自定义LLM规则 | 与大模型交互的prompt、格式、规则全部用户自定义 |
| 6 | 全透明 | LLM原始输入输出完全可见，不做封装、过滤、隐藏 |
| 7 | 前端化易安装 | 简单Web界面，一键启动，全球平台可分发 |
| 8 | 零依赖复制即用 | 无外部pip包依赖，全本地相对路径，复制文件夹即可运行 |
| 9 | 全中文命名 | 函数、变量、文件夹、界面全中文，方便中国人开发 |
| 10 | 全局/局部双层 | 全局命令+全局事件中心，局部文件挂载模块，局部事件后期引入 |
| 11 | 运行时高度实时 | 界面与大模型联动，后期可实时操作电脑 |
| 12 | 双引擎自我进化 | 主引擎稳定+工作引擎开发，人决定何时合并 |
| 13 | 进化安全红线 | 主引擎不可自改，权限不可自改，检测不过不合并，失败自动暂停 |
| 14 | 文件安全铁律 | 操作新文件夹必须问用户，无权限绝对不动，3次放行可升永久，权限存JSON随时改 |
| 15 | 大模型全权决策 | 系统不自己设计思路，所有问题丢给大模型优化解答 |
| 16 | 数字资产原则 | 一切代码和数据都是数字资产，尊重每一行代码，不允许冗余 |
| 17 | 日志+Git铁律 | 每次开发写好日志，Git提交含日期时间+标题 |
| 18 | 隐私分区 | 隐私区域与公开区域严格区分，隐私不进发布包 |

## 五、核心流程（怎么跑起来的）

### 5.1 启动流程
```
1. 双击 启动.bat / 启动.sh
2. 启动器读取 ./公共区/配置/系统配置.json
3. 配置加载器加载所有JSON配置（含技能配置、MCP服务配置）
4. 模型直连器从 ./隐私区/我的配置/密钥.json 读取密钥，验证连通性
5. 文件管理器初始化权限校验
6. 运行诊断器初始化（错误自动记录+监控规则引擎）
7. 操作注册中心注册内置操作（120个）
8. 动态工具加载器从 工具声明.json 加载声明式工具
9. 插件加载器扫描 ./公共区/插件/ 热加载插件
10. 技能加载器扫描 ./公共区/技能/ 解析SKILL.md+包装脚本
11. MCP管理器从 MCP服务.json 连接外部MCP Server，发现工具注册
12. 模块调度器扫描 ./公共区/模块/，按配置加载启用的模块
13. 定时任务调度器启动
14. 网页服务启动，监听端口（默认8765）
15. 浏览器自动打开 http://localhost:8765
```

### 5.2 五层工具体系

系统所有工具统一注册到操作注册中心，LLM 无感知差异，通过 Function Calling 调用：

| 层 | 来源 | 加载方式 | 数量 |
|----|------|----------|------|
| ① 内置操作 | 内核/操作/*.py | 启动器硬编码注册 | 120 |
| ② 动态工具 | 配置/工具声明.json | JSON声明→自动生成操作类 | 按需 |
| ③ 插件 | 公共区/插件/*.py | importlib动态加载 | 按需 |
| ④ 技能脚本 | 公共区/技能/*/scripts/ | 技能加载器包装为操作 | 按需 |
| ⑤ MCP工具 | 外部MCP Server | JSON-RPC发现→包装为操作 | 按需 |

### 5.3 对话流程
```
1. 用户输入消息
2. 对话模块打包上下文（用户画像+相关记忆+当前对话+文件环境+匹配经验）
3. 检查function_calling配置：
   - 启用：从操作注册中心获取工具定义，注入请求体tools字段
   - 未启用：在系统提示词中教模型用JSON格式调用操作
4. 通过模型直连器发送HTTP请求（urllib，不用SDK）
5. 解析响应：
   - 优先检查tool_calls（原生function calling结构化数据）
   - 无tool_calls时回退文本解析（4层JSON提取策略）
6. 执行操作 → 观察结果 → 循环直到LLM给出最终文字回复
7. 文件修改操作后自动验证Python语法（.py文件）
8. 原始请求和响应写入 ./隐私区/我的日志/（全透明）
9. 返回结果给用户
10. 记忆模块记录到当前事件
```

## 九、安全红线（绝对不可违反）

1. **文件权限**：操作任何新文件夹必须问用户，无权限绝对不动
2. **禁止路径**：Windows/System32/AppData等绝对不可碰
3. **3次放行**：同一路径询问3次可自动升级永久，但用户可随时撤销
4. **主引擎保护**：大模型不可直接修改 `./公共区/` 代码，只能通过合并
5. **权限不可自改**：`./公共区/配置/文件权限.json` 不允许大模型自行修改
6. **进化禁止范围**：`启动器.py` 和 `文件权限.json` 禁止自我进化修改
7. **失败暂停**：连续失败3次自动暂停，需人工恢复
8. **隐私区隔离**：`./隐私区/` 绝不进发布包，绝不推送远程Git
9. **密钥掩码**：API Key在界面和日志中掩码显示

### EXE打包流程

### exe运行机制
- `打包入口.py` 为PyInstaller入口，处理 `sys._MEIPASS`（打包资源）与工作目录（用户数据）分离
- 首次运行自动释放公共区配置 + 创建隐私区目录结构
- 用户数据存放在exe同级目录，不随exe内部资源丢失

### 静默自动更新
```
用户启动系统
  → 3秒后后台静默检查更新源
  → ① Gitee API（国内优先）
  → ② GitHub API（国外备用）
  → ③ 自建服务器（兜底）
  → 任一源发现新版本 → 右上角🔄变⏳ → 自动下载zip覆盖公共区
  → 成功 → 变✅ → Toast提示"已更新，点击重启"
  → 失败 → 变🔄 → Toast提示"点击重试"
```

### 更新安全措施
- 覆盖前自动备份公共区到 `公共区_backup_YYYYMMDD_HHMMSS/`
- 只覆盖公共区，绝不碰隐私区
- 失败自动回滚（从备份恢复）
- 只保留最近1个备份

## 十五、路径规范（开发者必读）

### 15.1 路径规则
- **所有路径使用相对路径**，相对于项目根目录（本文档所在目录）
- **禁止硬编码绝对路径**（如 `D:/xxx`、`C:/Users/xxx`）
- **禁止使用用户目录**（如 `~/xxx`、`%USERPROFILE%/xxx`）
- 代码中使用 `pathlib.Path(__file__).parent` 定位自身，再相对引用
- 配置JSON中所有路径以 `./` 开头，表示项目根目录

### 15.2 代码中获取路径
```python
import pathlib

# 获取项目根目录（假设本文件在 公共区/内核/）
项目根目录 = pathlib.Path(__file__).parent.parent.parent
公共区 = 项目根目录 / "公共区"
隐私区 = 项目根目录 / "隐私区"
配置目录 = 公共区 / "配置"

# 读取配置
系统配置路径 = 配置目录 / "系统配置.json"
```

### 15.3 配置中引用路径
```json
{
  "日志目录": "./隐私区/我的日志/",
  "模块目录": "./公共区/模块/",
  "记忆库路径": "./隐私区/我的记忆/记忆库.json"
}
```

### 15.4 为什么用相对路径
- 复制到任何电脑、任何磁盘、任何目录都能直接运行
- 不依赖特定用户的目录结构
- 打包发布时路径不受影响
- 符合原则8：零依赖复制即用

## 十六、设计哲学

> **够用就行，越小越好。**
> **系统是手脚，大模型是大脑。**
> **一切都是数字资产，尊重每一行代码。**

> **所有路径相对引用，复制到哪里都能跑。**

## 十七、故障排查

### 17.1 启动崩溃：UnicodeEncodeError 'gbk'

**现象**：`py "公共区/内核/启动器.py"` 直接报 `UnicodeEncodeError: 'gbk' codec can't encode character '\U0001f4cb'`

**原因**：Windows控制台默认GBK编码，print中的emoji字符（📋🤖🔒⚡等）无法编码

**修复**：启动器.py 文件头已加 `sys.stdout/stderr.reconfigure(encoding='utf-8', errors='replace')`，无需再改

### 17.2 界面"没反应"：浏览器缓存旧JS

**现象**：发消息后界面卡住，但后端API正常（`curl` 或 `urllib` 测试有回复）

**原因**：浏览器缓存了旧版本的 `逻辑.js` / `编辑器.js`，新代码不生效

**修复**：
1. `主页.html` 中JS引用加了版本号 `?v=20260619`
2. `网页服务.py` 的 `_返回文件` 加了 `Cache-Control: no-cache` 头
3. 用户按 **Ctrl+F5** 强制刷新浏览器

### 17.3 快速诊断流程

```
界面没反应？
├── 服务器在跑吗？ → netstat -ano | findstr "8765.*LISTEN"
│   ├── 没有 → py "公共区/内核/启动器.py" 重启
│   └── 有 → 下一步
├── API有响应吗？ → py -c "import urllib.request,json; ..." 测试 /api/chat
│   ├── 没响应 → 查日志/重启
│   └── 有响应 → 前端问题 → Ctrl+F5 强制刷新浏览器
```

## 十八、双文件夹开发与发布流程（重要）

### 18.1 两套文件夹的关系

| | 开发版 | 发布版 |
|---|--------|--------|
| **路径** | `<项目根目录>/新系统_v2_开发版` | `<项目根目录>/新系统_v2` |
| **用途** | 日常开发、调试、测试 | 对外发布、推送到GitHub |
| **隐私区** | ✅ 含真实数据（密钥/API Key/对话记录/记忆库/日志/浏览器会话） | ✅ 仅有空模板，无真实隐私数据 |
| **公共区** | ✅ 代码和配置（开发中最新） | ✅ 代码和配置（从开发版同步） |
| **git仓库** | ✅ 有 `.git`，**仅本地**，永远不配置远程 | ✅ 有 `.git`，远程为 `https://github.com/zf3dzf3dzf3d-ctrl/zf3d_agent.git` |
| **角色** | AI 的工作台，随便改 | AI 负责同步代码并推送，绝不碰隐私数据 |

### 18.2 核心原则

1. **开发版是AI的工作区**：AI在开发版中开发功能、修bug、调试
2. **发布版是干净版本**：只含公共区代码，不含任何密钥/对话/记忆等隐私数据
3. **开发版只有本地git**：开发版有 `.git` 用于追踪修改历史，但**永远不配置远程仓库**（`git remote` 为空），物理上无法 `push`。`.gitignore` 仍排除隐私数据，双重保险
4. **AI负责同步**：开发完成后，AI将公共区代码从开发版复制到发布版
5. **AI负责推送**：同步完成后，AI在发布版目录执行git提交并推送到GitHub
6. **隐私数据零泄露**：`.gitignore` 已排除所有隐私文件，但AI每次推送前仍需检查 staged 文件列表，确认无隐私文件
7. **两个git完全独立**：开发版和发布版各自有独立的 `.git`，互不感知、互不影响。开发版无远程仓库（`git remote` 为空），即使误执行 `git push` 也会报错。发布版的远程只指向 GitHub。父目录不可有 `.git`，避免误追踪两个子文件夹

### 18.3 同步与推送完整流程

```
开发版（新系统_v2_开发版）           发布版（新系统_v2）
┌──────────────────────┐           ┌──────────────────────┐
│  公共区/  ← 代码修改  │           │  公共区/  ← 同步代码  │
│  隐私区/  ← 真实数据  │           │  隐私区/  ← 空模板    │
│  .git（仅本地）       │           │  .git（连GitHub）     │
└──────────┬───────────┘           └──────────┬───────────┘
           │                                  │
           │  ① 对比差异（只看公共区文件）      │
           │  ② 复制修改的公共区文件 ─────────→ │
           │  ③ 哈希校验确认一致               │
           │                                  │
           │  ④ git add . && git commit       │  ⑤ git add . && git status 检查
           │     （本地保存，无法push）         │  ⑥ git commit -m "[类型] 时间 标题"
           │                                  │  ⑦ git push origin main
```

### 18.4 AI操作步骤（每次发布必跟）

**第1步：对比差异**
- 用工具对比两个版本 `公共区/` 目录下的文件
- 找出新增、修改、删除的文件
- `隐私区/` 永远不在对比范围内

**第2步：复制代码文件**
- 只复制 `公共区/` 下被修改的文件
- 如果有新增文件（如新的 .py / .json / .md），也要复制
- 绝对不复制 `隐私区/` 下任何文件

**第3步：哈希校验**
- 对每个复制的文件做 SHA256 校验，确认开发版和发布版内容一致

**第4步：检查git状态**
- 在发布版目录执行 `git add .` 然后 `git status`
- **逐条检查 staged 文件列表**：确认全是 `公共区/` 路径，无 `隐私区/` 文件
- 如发现隐私文件被 staged，立即 `git reset HEAD <文件>` 撤销

**第5步：提交**
```bash
cd "<发布版根目录>"
git commit -m "[同步] YYYY-MM-DD HH:mm 从开发版同步N个文件到发布版（简要说明改动）"
```

**第6步：推送到GitHub**
```bash
git push origin main
```

**第7步：确认推送成功**
- 检查命令输出无报错
- 推送成功后会显示 `main -> main` 的 commit hash 变化

### 18.5 .gitignore 隐私保护清单

发布版 `.gitignore` 已排除以下隐私内容（AI无需修改，但需了解）：

| 类型 | 排除规则 | 说明 |
|------|----------|------|
| 密钥 | `隐私区/我的配置/密钥.json` | API Key等，绝对不可提交 |
| 询问记录 | `隐私区/我的配置/询问记录.json` | 个人权限授权记录 |
| 对话记录 | `隐私区/对话记录/*.json` | 全部历史对话 |

| 记忆库 | `隐私区/我的记忆/记忆库.json` 等 | 个人记忆数据 |
| 运行日志 | `隐私区/我的日志/**` | 含LLM原始请求响应 |

| 开发日志 | `开发日志.md` | 私有版本，不发布 |
| Python缓存 | `__pycache__/` `*.pyc` | 编译产物 |

### 18.6 推送前安全检查清单

每次 `git push` 前必须确认：

- [ ] `git status` 显示的文件全在 `公共区/` 路径下
- [ ] 没有任何 `隐私区/` 文件出现在 staged 列表中
- [ ] 没有 `密钥.json`、`询问记录.json`、`记忆库.json` 等隐私文件
- [ ] commit message 包含时间（格式：YYYY-MM-DD HH:mm）
- [ ] commit message 准确描述本次改动内容
